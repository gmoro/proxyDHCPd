## 2024-05-24 - DoS via Unhandled Out-Of-Bounds Exception in Packet Parsing
**Vulnerability:** In `pydhcplib`'s packet parsing logic, `DecodePacket` did not check if the iterator was at the end of the packet data before attempting to read the length byte of a DHCP option (`iterator+1`). A specially crafted packet terminating exactly at an option byte code would throw an `IndexError: list index out of range`, potentially crashing the ProxyDHCP daemon handling the packet.
**Learning:** This existed because the original `pydhcplib` codebase assumed a well-formed network payload and blindly relied on `self.packet_data[iterator+1]`.
**Prevention:** Ensure all binary network data parsing functions bounds-check their read iterators against the maximum buffer length before consuming dynamically-sized tokens.
## 2025-02-28 - DoS via Unhandled Out-Of-Bounds Exception on Empty Options Array in Packet Parsing
**Vulnerability:** In `pydhcplib`'s `DhcpPacket.GetHardwareAddress()`, if the DHCP packet's `hlen` (Hardware Address Length) field returns an empty array when retrieved via `self.GetOption("hlen")` (e.g. from an artificially short network payload), it would result in an `IndexError` when attempting to access `[0]` on the list, crashing the ProxyDHCP service.
**Learning:** Network-parsing modules need to consider that fields populated natively by standard parsing rules may end up unexpectedly empty if the provided packet data lacks enough byte sequence to complete the parsing. Default bounds-check/presence checks are required even for standard packet headers.
**Prevention:** Avoid blindly indexing the first element (`[0]`) of array-like fields retrieved from packet structure parsing without ensuring the array actually has content. Check for array truthiness before indexing.
