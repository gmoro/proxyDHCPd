## 2024-05-24 - DoS via Unhandled Out-Of-Bounds Exception in Packet Parsing
**Vulnerability:** In `pydhcplib`'s packet parsing logic, `DecodePacket` did not check if the iterator was at the end of the packet data before attempting to read the length byte of a DHCP option (`iterator+1`). A specially crafted packet terminating exactly at an option byte code would throw an `IndexError: list index out of range`, potentially crashing the ProxyDHCP daemon handling the packet.
**Learning:** This existed because the original `pydhcplib` codebase assumed a well-formed network payload and blindly relied on `self.packet_data[iterator+1]`.
**Prevention:** Ensure all binary network data parsing functions bounds-check their read iterators against the maximum buffer length before consuming dynamically-sized tokens.
## 2024-05-24 - [Multiple Vulnerabilities: OOB Read, DoS, Insecure Permissions, Partial Match]
**Vulnerability:** Found multiple issues: `os.umask(0)` created world-writable files for root daemon, `GetHardwareAddress` could throw `IndexError` on malformed empty `hlen` option list, `DecodePacket` could read out of bounds on long option lengths, and `ipAddressCheck` allowed partial string matches.
**Learning:** Embedded protocol parsers (like pydhcplib) often lack robust bounds checking and empty-list handling for specific options. Daemonization routines need explicit secure `umask` settings. Regex `match` only checks the start of strings, not full matching.
**Prevention:** Always check if a parsed list is empty before accessing its elements. Verify `opt_len` bounds against buffer length. Always use `0o022` or similar secure umask when decoupling processes. Use `re.fullmatch` for complete string validation.
